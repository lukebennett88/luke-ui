Luke UI themes are static stylesheets. A theme root establishes a themed boundary, an identity
provides its visual foundation, and a colour mode selects its light or dark values. Components keep
the same API in every combination.

## How a themed boundary works

A theme root carries `themeRootClassName` and one identity class. Its descendants inherit the
identity's semantic tokens. Add `data-color-mode` to select a mode for the root or a smaller scope
inside it.

1. The theme root supplies the Luke UI reset and base theme styles.
2. The identity supplies semantic colour, typography, spacing, radii, material, and depth.
3. A colour mode selects the identity's light or dark token values.

The setup guide shows the CSS imports and root markup. This page explains how to arrange those
parts.

## Choose an identity

Luke UI ships with two bundled identities:

- Tactile is the default. It uses stronger depth and a compact material treatment.
- Paper is flatter, with lighter borders and depth.

Apply the matching CSS file and identity class at the same themed root as `themeRootClassName`. See
[Applying a theme](/theming/applying) for the exact imports and root markup.

Use one identity for an application or independent themed subtree. Identity classes are not
nestable, so a child scope cannot switch from Tactile to Paper without becoming a separate themed
boundary.

## Choose a colour mode

Each identity provides light and dark values. Omit `data-color-mode` to follow
`prefers-color-scheme`, or set it to `light` or `dark` to force a mode. Modes can nest within an
identity, which is useful for previews and intentional light-on-dark or dark-on-light boundaries.

## Keep the theme through portals

Application-owned portals render outside the themed DOM branch, so they cannot inherit its tokens.
Copy the nearest theme identity class and explicit colour-mode attribute to the portal root. When
the source follows the system setting, omit `data-color-mode` from the portal root too.

Luke UI's Combobox already does this for its popover. See [Applying a theme](/theming/applying) for
the complete portal contract.

## Theme values and component variants

Themes supply the values behind a component. Components expose the safe choices that use those
values. For example, Button uses `tone` and `appearance` to communicate intent and hierarchy, while
the theme supplies the final colours, finish, and depth.

```tsx
<Button appearance="solid" tone="accent">
	Save changes
</Button>
```

Use a component API before custom CSS. For a custom element, use the public `vars` token contract.
Do not depend on a component's selectors, generated palette values, or theme implementation details.

## Author a custom identity

When neither bundled identity fits a product, define a typed `ThemeFoundation` and compile it with
`buildTheme`. The generated stylesheet contains the full semantic contract for both colour modes and
validates contrast before it is written.

Read [Authoring a theme](/theming/authoring) for the foundation API and build step.
